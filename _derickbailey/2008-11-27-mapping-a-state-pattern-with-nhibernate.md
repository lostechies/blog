In the comments of my previous post &#8211; <a href="http://www.lostechies.com/blogs/derickbailey/archive/2008/11/26/ptom-descriptive-state-enumeration.aspx" target="_blank">Descriptive State Enumeration</a> &#8211; Maxim Tihobrazov asked me to show how to map a state pattern with NHibernate; and I am more than happy to oblige!

### NHibernate Mapping Options

I certainly don&#8217;t claim to be an expert on NHibernate, but I do use it on a daily basis and I&#8217;ve solved my fair share of problems &#8211; including how to map a State pattern with NHibernate. According the NHibernate <a href="http://www.hibernate.org/hib_docs/nhibernate/html/inheritance.html" target="_blank">inheritance documentation</a>, there are three core ways of mapping this pattern:

  1. Table per class hierarchy (I&#8217;ve always called it &#8220;record per class&#8221;) &#8211; each subclass (state instance, in this case) is associated with a specific record in a table. All of the subclasses are found in the same table.
  2. Table per subclass (also called a &#8220;joined sub class&#8221;) &#8211; each sub class has it&#8217;s own table and is joined to the super class by a one to one relationship with it&#8217;s primary key.
  3. Table per concrete class &#8211; each sub class has it&#8217;s own table and is mapped by the specific class type, not by the abstraction.

### My Choice &#8211; Table Per Class Hierarchy

In my current set of applications, I&#8217;ve always used the table per class hierarchy option so that is what I will describe here. My reasons for choosing this option are simplicity in the mapping and more importantly &#8211; not having a need to store the records in their own tables. Each of my states can be very cleanly represented as a record in my state table.

### Setting Up Shop

Let&#8217;s consider the same coffee shop that I&#8217;ve used in both of my PTOM posts &#8211; in this case, we&#8217;re dealing specifically with the Order class and it&#8217;s OrderStatus state. It&#8217;s necessary to note that when I am working with NHibernate and the state pattern, I actually do set up my state models a little different. First off &#8211; we need an Id for NHibernate to map to the table&#8217;s primary key. Secondly, I like to keep the core information in the abstract base class and provide all the varying values via a constructor. Third, each of the inheriting classes needs to provide a default (no args) constructor. Forth, we need a way for NHibernate to know which class it should actually instantiate, when loading &#8211; a &#8220;descriminator&#8221;. I like to use a simple string Name property for this. And lastly &#8211; I think (though I am likely wrong on this aspect) that the abstract base class needs to provide a default constructor. Fortunately, the constructors we need don&#8217;t need to be public &#8211; they can be private.

The key to all of this is the DescriminateSubClassesOnColumn method. The generics <string> tells us what .NET Type is being used to identify the specific class instances that we are dealing with and the (&#8220;Name&#8221;) parameter is the column name in the database that represents the specific instance. 

Then, we have the specific class instances referenced by the &#8220;.Subclass<type>&#8221; calls. This is a fluent interface, so &#8220;.Subclass&#8221; is a method that gets called directly off the DescriminatoeSubClassesOnColumn method. The specific type we want NHibernate to know about is specified in the generics parameter: &#8220;.SubClass<InProcessStatus>&#8221;. The &#8220;IsIdentifiedBy&#8221; is where we give NHibernate the knowledge of what value in the &#8220;Name&#8221; column maps to what specific class in our code. To prevent <a href="http://en.wikipedia.org/wiki/Magic_string_(programming)" target="_blank">magic string</a> syndrome from setting in, I like to use the actual enumeration pattern of my OrderStatus to specify the string name of the class. 

And finally, we have to provide an ugly workaround for our maps via the &#8220;MapSubclassColumns&#8221; method. There is an implementation issue in FluentNHibernate currently, and because of this issue we are forced to call the &#8220;MapSubclassColumns&#8221; method, with an empty lambda expression. If we don&#8217;t call this method, the sub class will not get registered and NHibernate will not know how to handle the data in question. (I am hoping to fix this issue at some time, and make it so we don&#8217;t have to call that method. I just haven&#8217;t had time recently.)

### Wrapping Up The Map

The rest of the map (the one remaining &#8220;name&#8221; property being mapped, in this case) is a regular NHibernate map. You&#8217;ll notice, though, that we never mapped the &#8220;DispalyForFullfillment&#8221; property. I explicitly choose to leave any and all &#8220;volatile&#8221; properties out of my state maps. By doing this, I am able to add, edit, and remove any properties or methods on the state objects that I need, without having to change the NHibernate mappings. Since the state objects I am dealing with don&#8217;t change without recompiling the code anyway, I don&#8217;t need the ability to define them in the database. However, if you do need or want the flexibility of defining your states in the database, you can map each individual property of the state. Just remember that you will have to modify both the code and the database, to make the changes complete. 

I hope this quick look at mapping a state pattern with Fluent NHibernate will help to shed some light on the subject, for someone. I realize that I have only provided the FluentNHibernate mapping, though. I specifically chose to do this because it would be a serious chore for myself to create a code project with NHibernate set up so that I can actually verify my mapping xml is correct. However, the translation from FluentNHibernate back to standard .hbm.xml files should be fairly straightforward, with the help of the <a href="http://www.hibernate.org/hib_docs/nhibernate/html/" target="_blank">NHibernate Documentation</a>.
